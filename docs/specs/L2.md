# L2 - Detailed Requirements

## L2-001: Register a new AI agent
**Traces to:** L1-001

The system must allow users to register an AI agent by providing a unique name, a type (Claude, Gemini, Copilot, Codex, or Custom), an optional executable path, and optional default parameters. The agent must be stored in an in-memory registry for subsequent retrieval and invocation.

**Acceptance Criteria:**
1. Given a unique agent name and valid type, when the user calls `Register-AIAgent`, then the agent must be stored in the registry and the AIAgent object must be returned.
2. Given no explicit executable path, when the user registers an agent, then the executable path must default to the agent type name (e.g., "claude" for Claude).
3. Given default parameters are provided, when the user registers an agent, then those parameters must be stored on the agent and merged at invocation time.

---

## L2-002: Detect agent executable availability
**Traces to:** L1-001

Upon registration, the system must test whether the agent's executable is available on the system PATH using `Get-Command` and set the agent's `IsAvailable` property accordingly.

**Acceptance Criteria:**
1. Given the executable exists on PATH, when the agent is registered, then `IsAvailable` must be set to `$true`.
2. Given the executable does not exist on PATH, when the agent is registered, then `IsAvailable` must be set to `$false` and registration must still succeed.

---

## L2-003: Prevent duplicate agent registration
**Traces to:** L1-001

The system must reject registration of an agent with a name that already exists in the registry, unless the `-Force` switch is provided to overwrite the existing entry.

**Acceptance Criteria:**
1. Given an agent named "MyAgent" already exists, when the user calls `Register-AIAgent -Name "MyAgent"` without `-Force`, then the system must throw a terminating error.
2. Given an agent named "MyAgent" already exists, when the user calls `Register-AIAgent -Name "MyAgent" -Force`, then the existing agent must be replaced with the new configuration.

---

## L2-004: Retrieve registered agents
**Traces to:** L1-001

The system must allow retrieval of a single agent by name or all registered agents as a collection.

**Acceptance Criteria:**
1. Given an agent named "Claude" is registered, when the user calls `Get-AIAgent -Name "Claude"`, then the corresponding AIAgent object must be returned.
2. Given multiple agents are registered, when the user calls `Get-AIAgent` with no name parameter, then all registered agents must be returned.
3. Given no agent with the requested name exists, when the user calls `Get-AIAgent -Name "NonExistent"`, then the system must throw a terminating error.

---

## L2-005: Remove registered agents
**Traces to:** L1-001

The system must allow removal of a registered agent by name, with support for `-WhatIf` and `-Confirm` parameters.

**Acceptance Criteria:**
1. Given an agent named "Claude" is registered, when the user calls `Remove-AIAgent -Name "Claude"`, then the agent must be removed from the registry.
2. Given no agent with the requested name exists, when the user calls `Remove-AIAgent -Name "NonExistent"`, then the system must throw a terminating error.
3. Given the user calls `Remove-AIAgent -Name "Claude" -WhatIf`, then no agent must be removed and the operation must be previewed.

---

## L2-006: Invoke a registered agent by name
**Traces to:** L1-002

The system must allow invocation of a registered agent by name, passing a prompt string and optional call-time parameters that are merged with the agent's default parameters.

**Acceptance Criteria:**
1. Given a registered agent named "Claude", when the user calls `Invoke-AIAgent -Name "Claude" -Prompt "Hello"`, then the system must dispatch the prompt to the Claude CLI and return the text response.
2. Given the agent has default parameters and call-time parameters are provided, when the agent is invoked, then call-time parameters must be merged with default parameters.
3. Given the agent's CLI returns a non-zero exit code, when the agent is invoked, then the system must throw a terminating error.

---

## L2-007: Invoke an agent by type without registration
**Traces to:** L1-002

The system must allow one-off invocation of an agent by specifying a type (and optional executable path) without requiring prior registration.

**Acceptance Criteria:**
1. Given a valid agent type and prompt, when the user calls `Invoke-AIAgent -Type Claude -Prompt "Hello"`, then the system must create a temporary agent and dispatch the prompt.
2. Given a Custom type with an executable path, when the user calls `Invoke-AIAgent -Type Custom -ExecutablePath "./myagent" -Prompt "Hello"`, then the system must invoke the specified executable.

---

## L2-008: Agent-specific CLI dispatch
**Traces to:** L1-002

The system must dispatch prompts to each agent type using type-specific CLI argument conventions:
- **Claude:** `claude -p <prompt> [--key value ...]`
- **Gemini:** `gemini [--key value ...] <prompt>`
- **Copilot:** `gh copilot suggest -t <target> [--key value ...] <prompt>` (target defaults to "shell")
- **Codex:** `codex [--key value ...] <prompt>` (api_key parameter filtered from arguments)
- **Custom:** `<ExecutablePath> [--key value ...] <prompt>`

**Acceptance Criteria:**
1. Given a Claude agent, when invoked with prompt "analyze this" and parameter `model`="opus", then the CLI must be called as `claude --model opus -p "analyze this"`.
2. Given a Copilot agent with no target parameter, when invoked, then the target must default to "shell" and CLI must include `-t shell`.
3. Given a Codex agent with an `api_key` parameter, when invoked, then the `api_key` must not appear in the CLI arguments.
4. Given a Custom agent without an ExecutablePath set, when invoked, then the system must throw a terminating error.

---

## L2-009: Create a new workflow
**Traces to:** L1-003

The system must allow users to create a named workflow with an optional description, workflow-level variables, and an optional log path. The workflow must be stored in an in-memory registry with UTC creation and update timestamps.

**Acceptance Criteria:**
1. Given a unique workflow name, when the user calls `New-Workflow -Name "Pipeline"`, then a Workflow object must be created, stored in the registry, and returned.
2. Given variables are provided, when the workflow is created, then the variables must be stored on the workflow for template resolution at execution time.
3. Given a LogPath is provided, when the workflow is created, then the LogPath must be stored and used during execution for file-based logging.

---

## L2-010: Prevent duplicate workflow creation
**Traces to:** L1-003

The system must reject creation of a workflow with a name that already exists in the registry, unless the `-Force` switch is provided.

**Acceptance Criteria:**
1. Given a workflow named "Pipeline" already exists, when the user calls `New-Workflow -Name "Pipeline"` without `-Force`, then the system must throw a terminating error.
2. Given a workflow named "Pipeline" already exists, when the user calls `New-Workflow -Name "Pipeline" -Force`, then the existing workflow must be replaced.

---

## L2-011: Retrieve and remove workflows
**Traces to:** L1-003

The system must allow retrieval of a single workflow by name or all workflows, and removal of a workflow by name with `-WhatIf`/`-Confirm` support.

**Acceptance Criteria:**
1. Given a workflow named "Pipeline" exists, when the user calls `Get-Workflow -Name "Pipeline"`, then the Workflow object must be returned.
2. Given multiple workflows exist, when the user calls `Get-Workflow` with no name, then all workflows must be returned.
3. Given no workflow with the requested name exists, when the user calls `Get-Workflow -Name "Missing"`, then the system must throw a terminating error.
4. Given a workflow named "Pipeline" exists, when the user calls `Remove-Workflow -Name "Pipeline"`, then it must be removed from the registry.

---

## L2-012: Add steps to a workflow
**Traces to:** L1-004

The system must allow users to add steps to a workflow, each with a unique step name, a reference to a registered agent, a prompt template, and optional configuration for dependencies, parameters, execution mode, retries, timeout, and callbacks.

**Acceptance Criteria:**
1. Given a workflow exists, when the user calls `Add-WorkflowStep` with a unique step name, agent name, and prompt template, then the step must be added to the workflow and the WorkflowStep object must be returned.
2. Given a step name already exists in the workflow, when the user calls `Add-WorkflowStep` with that name, then the system must throw a terminating error.
3. Given the `-Parallel` switch is provided, when the step is added, then its ExecutionMode must be set to Parallel.
4. Given MaxRetries is set to a value between 0 and 10, when the step is added, then the retry count must be stored on the step.
5. Given TimeoutSeconds is set to a value between 1 and 3600, when the step is added, then the timeout must be stored on the step (default: 120).
6. Given OnSuccess and OnFailure scriptblocks are provided, when the step is added, then the callbacks must be stored on the step.

---

## L2-013: Add steps via pipeline input
**Traces to:** L1-004

The system must accept a Workflow object via pipeline input when adding steps, in addition to accepting a workflow name string.

**Acceptance Criteria:**
1. Given a workflow object, when it is piped to `Add-WorkflowStep`, then the step must be added to that workflow.
2. Given a workflow name string, when `-WorkflowName` is used, then the step must be added to the workflow resolved from the registry.

---

## L2-014: Execute a workflow sequentially
**Traces to:** L1-005

The system must execute workflow steps in their defined order. For sequential steps, each step must complete before the next begins. The system must produce a WorkflowResult containing per-step results, overall success/failure, timing, and errors.

**Acceptance Criteria:**
1. Given a workflow with three sequential steps, when `Invoke-Workflow` is called, then steps must execute in order and the final step's output must be returned.
2. Given `-PassThru` is specified, when the workflow completes, then a WorkflowResult object must be returned containing StepResults, Success, Errors, Duration, and Output.
3. Given `-PassThru` is not specified, when the workflow completes successfully, then only the final output string must be returned.

---

## L2-015: Halt workflow on step failure
**Traces to:** L1-005

When any step fails (after exhausting retries), the workflow must halt immediately. Remaining steps must not execute. The WorkflowResult must reflect failure with accumulated errors.

**Acceptance Criteria:**
1. Given a workflow with steps A, B, C where step B fails, when the workflow is executed, then step C must not execute and the WorkflowResult.Success must be `$false`.
2. Given a step fails, when the workflow halts, then the error message must be included in WorkflowResult.Errors.

---

## L2-016: Execute parallel step groups concurrently
**Traces to:** L1-006

The system must group consecutive parallel-mode steps together and execute them concurrently using PowerShell background jobs. Each parallel step must run as an independent job.

**Acceptance Criteria:**
1. Given steps B and C are both marked as Parallel and are consecutive, when the workflow executes, then B and C must run concurrently as background jobs.
2. Given a sequence [A(seq), B(par), C(par), D(seq)], when the workflow executes, then A runs first, then B and C run concurrently, then D runs after both B and C complete.
3. Given parallel steps complete, when the next sequential step begins, then the last parallel step's output must be available as `{PreviousOutput}`.

---

## L2-017: Resolve prompt template tokens
**Traces to:** L1-007

The system must resolve prompt templates by replacing tokens enclosed in curly braces with values from the execution context.

**Acceptance Criteria:**
1. Given a template containing `{Input}`, when resolved, then the token must be replaced with the workflow's input data string.
2. Given a template containing `{PreviousOutput}`, when resolved, then the token must be replaced with the output of the most recently completed step.
3. Given a template containing `{StepName.Output}`, when resolved, then the token must be replaced with the output of the named step.
4. Given a template containing `{VariableName}` matching a workflow variable, when resolved, then the token must be replaced with the variable's value.
5. Given a template containing `{Env.VAR_NAME}`, when resolved, then the token must be replaced with the value of the environment variable `VAR_NAME`.
6. Given a template containing a token that does not match any context key or environment variable, when resolved, then the token must be left unchanged in the output.

---

## L2-018: Variable precedence and merging
**Traces to:** L1-007

When resolving templates, call-time variables provided to `Invoke-Workflow` must override workflow-level variables defined at creation time.

**Acceptance Criteria:**
1. Given a workflow with variable `Lang=Python` and call-time variable `Lang=Go`, when the template `{Lang}` is resolved, then the result must be "Go".
2. Given a workflow variable `Foo=Bar` and no call-time override, when the template `{Foo}` is resolved, then the result must be "Bar".

---

## L2-019: Retry on agent invocation failure
**Traces to:** L1-008

The system must retry failed agent invocations up to the configured MaxRetries count. For `Invoke-AIAgent`, the delay between retries must be configurable (1-300 seconds, default 5). For workflow steps, the inter-retry delay must be 3 seconds.

**Acceptance Criteria:**
1. Given `Invoke-AIAgent` is called with `-MaxRetries 2 -RetryDelaySeconds 10`, when the first attempt fails and the second succeeds, then the output from the second attempt must be returned with a 10-second delay between attempts.
2. Given a workflow step with MaxRetries=3, when all 4 attempts fail, then the step must be marked as failed and the workflow must halt.
3. Given a workflow step with MaxRetries=1, when the first attempt fails and the retry succeeds, then the step must be marked as successful.

---

## L2-020: Timeout for parallel steps
**Traces to:** L1-008

Parallel steps must enforce a configurable timeout (1-3600 seconds, default 120). If a parallel step's background job exceeds the timeout, the job must be stopped and the step must be marked as failed.

**Acceptance Criteria:**
1. Given a parallel step with TimeoutSeconds=5, when the background job runs for 10 seconds, then the job must be stopped and the step result must indicate failure due to timeout.
2. Given a parallel step with default timeout, when the background job completes within 120 seconds, then the step result must reflect success.

---

## L2-021: Lifecycle callbacks on step completion
**Traces to:** L1-004

The system must invoke OnSuccess and OnFailure scriptblock callbacks after each step completes, passing the StepResult as an argument.

**Acceptance Criteria:**
1. Given a step with an OnSuccess callback, when the step completes successfully, then the OnSuccess scriptblock must be invoked with the StepResult.
2. Given a step with an OnFailure callback, when the step fails, then the OnFailure scriptblock must be invoked with the StepResult.
3. Given a step with no callbacks defined, when the step completes, then no callback invocation must occur and execution must continue normally.

---

## L2-022: Export workflow to JSON
**Traces to:** L1-009

The system must serialize a workflow and its steps to a JSON file, including all step configuration except OnSuccess/OnFailure scriptblock callbacks, which cannot be serialized.

**Acceptance Criteria:**
1. Given a workflow with steps, when `Export-Workflow` is called, then a JSON file must be written containing the workflow name, description, variables, log path, creation timestamp, and all step definitions.
2. Given the output file already exists and `-Force` is not specified, when `Export-Workflow` is called, then the system must throw a terminating error.
3. Given the output file already exists and `-Force` is specified, when `Export-Workflow` is called, then the file must be overwritten.
4. Given a step has OnSuccess/OnFailure callbacks, when the workflow is exported, then the callbacks must be omitted from the JSON output.

---

## L2-023: Import workflow from JSON
**Traces to:** L1-009

The system must deserialize a workflow from a JSON file and register it in the workflow registry, reconstructing all step objects with their configuration.

**Acceptance Criteria:**
1. Given a valid workflow JSON file, when `Import-Workflow` is called, then the workflow must be reconstructed and registered, and the Workflow object must be returned.
2. Given the JSON file does not exist, when `Import-Workflow` is called, then the system must throw a terminating error.
3. Given a workflow with the same name already exists in the registry, when `Import-Workflow` is called without `-Force`, then the system must throw a terminating error.
4. Given a workflow with the same name already exists and `-Force` is specified, when `Import-Workflow` is called, then the existing workflow must be replaced.
5. Given an exported workflow contained OnSuccess/OnFailure callbacks, when imported, then those callbacks must not be present on the imported steps.

---

## L2-024: Structured logging at multiple severity levels
**Traces to:** L1-010

The system must log all operations with UTC timestamps in the format `[yyyy-MM-dd HH:mm:ss.fff] [LEVEL] message`. Log entries must be routed to the appropriate PowerShell stream based on severity level.

**Acceptance Criteria:**
1. Given an INFO-level log entry, when logged, then it must be written to the Verbose stream via `Write-Verbose`.
2. Given a WARN-level log entry, when logged, then it must be written to the Warning stream via `Write-Warning`.
3. Given an ERROR-level log entry, when logged, then it must be written to the Error stream via `Write-Error` with `-ErrorAction Continue`.
4. Given a DEBUG-level log entry, when logged, then it must be written to the Debug stream via `Write-Debug`.

---

## L2-025: File-based workflow execution logging
**Traces to:** L1-010

When a workflow has a LogPath configured, all log entries generated during execution must be appended to the specified file with UTF-8 encoding. File write failures must not halt execution.

**Acceptance Criteria:**
1. Given a workflow with LogPath set to "output.log", when the workflow executes, then all log entries must be appended to "output.log".
2. Given the log file write fails (e.g., permissions), when a log entry is written, then a warning must be emitted but execution must continue without interruption.

---

## L2-026: Mandatory parameter validation
**Traces to:** L1-011

The system must validate all mandatory parameters and reject calls with missing or invalid values using PowerShell's parameter validation attributes.

**Acceptance Criteria:**
1. Given `Register-AIAgent` is called without `-Name`, when the command is parsed, then PowerShell must reject the call with a parameter binding error.
2. Given `Add-WorkflowStep` is called with `-MaxRetries 15` (exceeds range 0-10), when the command is parsed, then PowerShell must reject the call with a validation error.
3. Given `Add-WorkflowStep` is called with `-TimeoutSeconds 0` (below range 1-3600), when the command is parsed, then PowerShell must reject the call with a validation error.

---

## L2-027: Descriptive error messages for invalid operations
**Traces to:** L1-011

The system must throw descriptive terminating errors when users attempt invalid operations such as referencing non-existent agents or workflows, or adding duplicate entries without the Force flag.

**Acceptance Criteria:**
1. Given no agent named "X" exists, when `Get-AIAgent -Name "X"` is called, then the error message must identify the agent name that was not found.
2. Given a workflow step references an unregistered agent, when the workflow is executed, then the error must identify which agent was not found.
3. Given a duplicate agent name is used without `-Force`, when `Register-AIAgent` is called, then the error must indicate the agent already exists.

---

## L2-028: Workflow input and output handling
**Traces to:** L1-005

The system must accept an input string for workflow execution (via `-WorkflowInput` or `-Input` alias) and make it available as the `{Input}` token in prompt templates. The system must track each step's output and make the final successful step's output available as the workflow's output.

**Acceptance Criteria:**
1. Given a workflow is invoked with `-WorkflowInput "analyze code"`, when the first step's template contains `{Input}`, then "analyze code" must be substituted.
2. Given a multi-step workflow completes successfully, when the result is returned without `-PassThru`, then the last step's output string must be the return value.
3. Given a workflow halts on step 2 of 3, when the result is returned with `-PassThru`, then WorkflowResult.Output must contain the last successful step's output.

---

## L2-029: Step output cross-referencing
**Traces to:** L1-007

The system must allow steps to reference outputs of previously completed steps by name using `{StepName.Output}` or the shorthand `{StepName}` syntax in prompt templates.

**Acceptance Criteria:**
1. Given step "Analyze" completed with output "result A", when step "Summarize" has template "Summarize: {Analyze.Output}", then the resolved prompt must be "Summarize: result A".
2. Given step "Analyze" completed with output "result A", when step "Summarize" has template "Summarize: {Analyze}", then the resolved prompt must be "Summarize: result A".
3. Given step "Analyze" has not yet completed, when a prior step references `{Analyze.Output}`, then the token must be left unresolved.

---

## L2-030: Secure argument passing to CLI executables
**Traces to:** L1-012

The system must pass arguments to CLI executables using array-based argument lists (via the `&` call operator) rather than string concatenation to prevent shell injection.

**Acceptance Criteria:**
1. Given parameters containing shell metacharacters (e.g., `; rm -rf /`), when the agent is invoked, then the metacharacters must be treated as literal values and not interpreted by the shell.
2. Given all agent invocations, when CLI processes are started, then arguments must be passed as array elements to the `&` operator.

---

## L2-031: Exclude sensitive parameters from CLI arguments
**Traces to:** L1-012

The system must filter sensitive parameters (such as `api_key`) from CLI argument lists for agent types that expect credentials via environment variables rather than command-line flags.

**Acceptance Criteria:**
1. Given a Codex agent with an `api_key` parameter, when invoked, then `api_key` must not appear in the CLI arguments.
2. Given a Codex agent, when invoked, then the API key must be expected from the `OPENAI_API_KEY` environment variable.

---

## L2-032: Exclude scriptblock callbacks from serialized output
**Traces to:** L1-012

The system must not serialize OnSuccess/OnFailure scriptblock callbacks when exporting workflows to JSON to prevent code injection on import.

**Acceptance Criteria:**
1. Given a workflow step with OnSuccess and OnFailure callbacks, when exported to JSON, then neither callback must appear in the JSON output.
2. Given a workflow JSON file, when imported, then no arbitrary code execution must occur during deserialization.

---

## L2-033: Workflow execution timing and duration tracking
**Traces to:** L1-005

The system must track start time, end time, and duration for each step and for the overall workflow execution, recording these on StepResult and WorkflowResult objects.

**Acceptance Criteria:**
1. Given a workflow is executed, when the WorkflowResult is returned with `-PassThru`, then StartTime, EndTime, and Duration must all be populated.
2. Given a step completes, when its StepResult is examined, then StartTime, EndTime, and Duration must reflect the actual execution window.
3. Given a workflow with multiple steps, when WorkflowResult.Duration is checked, then it must equal the elapsed time from the first step's start to the last step's end.

---

## L2-034: PowerShell module structure and compatibility
**Traces to:** L1-011

The system must be packaged as a PowerShell module (.psd1/.psm1) compatible with PowerShell 5.1 and later, exporting exactly the public cmdlets and following PowerShell Verb-Noun naming conventions.

**Acceptance Criteria:**
1. Given the module manifest, when inspected, then it must declare PowerShellVersion = '5.1' as the minimum.
2. Given the module manifest, when FunctionsToExport is inspected, then it must list exactly: Register-AIAgent, Get-AIAgent, Remove-AIAgent, Invoke-AIAgent, New-Workflow, Add-WorkflowStep, Get-Workflow, Remove-Workflow, Invoke-Workflow, Export-Workflow, Import-Workflow.
3. Given the module is imported, when cmdlets are listed, then all exported functions must follow PowerShell-approved Verb-Noun naming conventions.
